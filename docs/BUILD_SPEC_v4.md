**Specification version:** 4.0

**Expected model version:** `atlas-v4`  
**Artifact-set identifier:** Read from `manifest.json`; never hard-code it in application logic.  
**Status:** Implementation contract

---

# 1. Product definition

NYC 311 Priority Atlas is an explorer-first urban-service intelligence workspace for investigating:

- mapped 311 complaint demand;
- per-capita complaint intensity;
- tract demographics;
- recorded administrative closure;
- spatial relationships among contiguous census tracts;
- deterministic prioritization assumptions;
- historical workload dynamics;
- transparent workload scenarios;
- evidence-grounded hypotheses generated by Claude.

The product is not:

- a chat-first application;
- a static map;
- a current NYC operations dashboard;
- a causal intervention model;
- an autonomous government recommendation system;
- a staffing, budget, or case-allocation optimizer.

The application must be complete and useful without Claude. Claude is an optional language and interpretation layer over deterministic application functions.

---

# 2. Product thesis

Traditional 311 dashboards answer where requests occurred. Priority Atlas extends that workflow by allowing a user to ask:

1. What complaint activity is concentrated in this tract?
2. Is the tract unusual relative to nearby tracts?
3. Which complaint domains and complaint types drive the pattern?
4. What does recorded administrative closure look like?
5. How does the result change under a different prioritization philosophy?
6. How did historical request cohorts arrive, age, close, and remain open?
7. What happens to modeled open workload under explicit demand and closure assumptions?
8. What evidence should be investigated before proposing an intervention?

The analytical system supplies all numbers. Claude explains, translates intent, and generates clearly labeled hypotheses.

# 3. Non-negotiable analytical boundaries

## 3.1 Historical scope

The request-arrival cohort is from 2016.

Use:

- `2016 historical snapshot`
- `2016 creation cohort`
- `historical period pattern`

Do not use:

- `current conditions`
- `today`
- `current agency performance`
- `forecast based on recent data`

## 3.2 Complaint semantics

311 records reflect both underlying conditions and reporting behavior. Complaint volume is not a direct measurement of physical need, harm, severity, service quality, resident awareness, or equal access to reporting.

## 3.3 Closure semantics

`Closed Date` is an administrative field.

Use:

- `recorded administrative closure`
- `recorded closed within 30 days`
- `recorded closed within 180 days`
- `not recorded closed by age 30 days`

Do not use:

- `resolved`
- `fixed`
- `successfully completed`
- `service delivered`
- `problem eliminated`

## 3.4 Scenario semantics

Scenario Lab generates deterministic selection scenarios.

Use:

- `selection scenario`
- `priority portfolio size`
- `priority balance`
- `compare assumptions`

Do not use:

- `recommended plan`
- `optimal allocation`
- `best policy`
- `agency capacity`
- `staffing plan`

## 3.5 Workload semantics

The historical replay is an age-structured reconstruction using actual 2016 arrival periods and recorded closure-by-age behavior. The scenario sandbox is an assumption-based projection.

Use:

- `historical replay`
- `expected open balance under historical closure behavior`
- `assumption-based scenario`
- `request-age checkpoint`
- `net open-workload change`

Do not use:

- `future forecast`
- `predicted agency backlog`
- `causal impact`
- `guaranteed reduction`
- `estimated staffing effect`

# 6. Application architecture

Use a single-page, map-centered application with three primary workspaces:

1. **Explore** — default
2. **Scenario Lab**
3. **Workload**

Comparison is embedded in Explore. Claude is available across all workspaces.

## 6.1 Desktop composition

- Full-viewport map canvas.
- Compact top navigation.
- Map controls at upper-left.
- Legend at lower-left.
- Right analytical rail:
  - about 420 px for one selected tract;
  - expandable to about 720 px or 55 vw for multi-tract comparison.
- Claude composer:
  - collapsed by default;
  - at the bottom of the analytical rail;
  - visually secondary;
  - expandable without replacing the map.

## 6.2 Mobile composition

- Full-screen map.
- Bottom sheet for details and comparison.
- Compact domain and metric controls.
- Claude as a separate bottom-sheet tab.
- No keyboard modifier required for selection.

## 6.3 Application state

Represent state explicitly:

```ts
type Workspace = "explore" | "scenario" | "workload";

interface AtlasState {
  workspace: Workspace;
  activeDomain: DomainKey;
  activeMapMetric: MapMetric;
  selectedGeoids: string[];
  activeGeoid: string | null;
  neighborhood: {
    enabled: boolean;
    radius: 1 | 2 | 3 | 4 | 5;
    metric: NeighborhoodMetric;
  };
  scenario: {
    currentScenarioId: string | null;
    pinnedScenarioId: string | null;
  };
  workload: {
    scope: WorkloadScope;
    demandChangePct: number;
    closureCurveShiftPoints: number;
    intervalLevel: 80 | 95;

    open: boolean;
    pendingAction: AssistantAction | null;
  };
}
```

Persist shareable analytical state in the URL: workspace, domain, metric, selected GEOIDs, active GEOID, neighborhood settings, scenario IDs, and workload assumptions. Do not place Claude conversation text in the URL.

# 12. Sparse tract-domain policy

Every one of the 10,835 tract-domain combinations exists.

```ts
type WorkloadSampleStatus =
  | "no_requests"
  | "no_known_timing"
  | "insufficient_sample"
  | "sufficient";
```

Definitions:

- `no_requests`: request count is zero.
- `no_known_timing`: requests exist, known timing is zero.
- `insufficient_sample`: known timing is between 1 and 29.
- `sufficient`: known timing is at least 30.

For the first three:

- no tract-specific closure rate in primary UI;
- no open-at-age estimate;
- no replay;
- no uncertainty.

Missing or suppressed values remain `null`, never zero.

Version 1 does not automatically substitute a borough, city, neighbor, or hierarchical curve.

Explicit group pooling is allowed for selected tracts, neighborhoods, and scenario selections. Aggregate by summing period arrays and closure counts. A pooled curve is usable only when combined known timing reaches 30. Label it `Pooled across N tracts`.

Median closure time is shown only when valid recorded closures reach 30.

# 15. Workload workspace

Two tabs:

1. Historical Replay
2. Scenario

## 15.1 Scope

- active tract;
- selected tracts;
- active neighborhood;
- current scenario;
- pinned scenario.

Display scope explicitly.

## 15.2 Aggregation

Sum:

- period arrivals elementwise;
- known timing;
- valid closures;
- closed-by-age elementwise.

Do not average tract percentages.

## 15.3 Historical period pattern

Show 13 actual periods.

- Periods 0–11: complete 30-day periods.
- Period 12: six days.
- Partial period must be visually distinct.
- Exclude partial period from mean, median, resampling, and uncertainty.

Label:

`2016 historical 30-day arrival pattern`

Do not call it seasonality or forecast.

## 15.4 Age-structured replay

Use:

\[
B_t = \sum_{c \le t} A_c S(\text{age}_{c,t})
\]

\[
\Delta B_t = B_t - B_{t-1}
\]

Each period contains:

- new requests;
- expected recorded closures;
- expected open balance;
- net open change;
- open by age.

Main chart:

- arrivals;
- expected closures;
- open balance.

Secondary chart:

- 0–30;
- 31–60;
- 61–90;
- 91–180;
- 181–360;
- 361+.

All expected values may be fractional.

## 15.5 Request-age checkpoint panel

Use:

`Evaluate a 30-day arrival cohort after: 30 days / 180 days`

Do not use `Projection horizon`.

Show cohort open estimate, closure median, 80% interval, and 95% interval in expanded details.

## 15.6 Uncertainty

For sufficient scopes:

1. sample one of 12 complete arrival totals with replacement;
2. sample closure probability from:
   \[
   p \sim \operatorname{Beta}(closed+0.5, known-closed+0.5)
   \]
3. calculate:
   \[
   U=A(1-p)
   \]
4. repeat 1,000 times;
5. report median, 80%, and 95%.

Use a deterministic seed derived from artifact set, sorted GEOIDs, domain, age, and assumptions. Run in a Web Worker.

Tooltip:

`This interval reflects variation across the twelve complete 2016 arrival periods and finite-sample uncertainty in recorded closure. It does not include all possible future structural change.`

## 15.7 Scenario tab

Demand change:

-30% to +50%

\[
A'_t=A_t(1+g)
\]

Closure-curve shift:

-15 to +15 percentage points

\[
p'_a=\operatorname{clamp}(p_a+\delta,0,1)
\]

Validate monotonicity after clamping.

Compare baseline and scenario for arrivals, closures, open balance, net change, age composition, cohort open at 30 and 180, and uncertainty.

Label:

`Assumption-based workload scenario`

No causal claim.

## 15.8 Sparse scope

When aggregate known timing is below 30:

- show arrival history and request counts;
- do not show closure curve;
- do not show replay;
- do not show open-at-age estimates;
- do not show uncertainty;
- explain that more pooled timing outcomes are required.

# 26. Positioning
